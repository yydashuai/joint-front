## 一、方案概述
### 1.1 核心目标
针对接入消息队列的业务系统，提供标准化的自动化探测能力，快速回答三个核心问题：
1. MQ中间件本身是否正常运行、可访问？
2. 被测业务系统作为生产者，能否正常向MQ发送消息？
3. 被测业务系统作为消费者，能否正常从MQ接收并确认消息？

### 1.2 设计原则
- **非侵入优先**：优先通过MQ原生管理接口、客户端API完成探测，尽量不修改被测业务系统代码。
- **配置化驱动**：所有测试任务通过配置定义，新增被测系统无需修改代码，只需新增配置。
- **多MQ兼容**：通过适配层屏蔽RabbitMQ、Kafka、RocketMQ的API差异，上层逻辑统一。
- **低业务干扰**：测试消息严格隔离，生产环境优先采用被动探测，避免影响正常业务。

## 二、整体架构设计
整体采用分层架构，与现有测试系统通过统一的任务调度、结果告警模块对接，架构分层如下：

```
┌─────────────────────────────────────────────────────┐
│  现有测试系统（任务入口 / 结果展示 / 告警中心）        │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│  测试任务管理层：任务配置、调度触发、结果聚合          │
├─────────────────────────────────────────────────────┤
│  测试执行引擎：三大维度测试逻辑 + 结果判定            │
├─────────────────────────────────────────────────────┤
│  MQ多协议适配层：统一抽象接口 + 各MQ驱动实现          │
├─────────────────────────────────────────────────────┤
│  数据存储层：测试记录、历史状态、告警日志              │
└─────────────────────────────────────────────────────┘
```

### 核心模块职责
1. **测试任务管理层**
    - 维护所有被测MQ实例、被测系统的配置信息（连接地址、认证、队列/主题、触发方式等）
    - 支持定时调度、手动触发两种执行模式
    - 聚合测试结果，对接现有系统的告警与展示能力

2. **测试执行引擎**
    - 核心业务逻辑层，分别实现Broker健康检查、生产端连通性、消费端连通性三类测试流程
    - 内置超时控制、重试机制、结果判定规则
    - 输出标准化的测试结果：成功/失败/警告、失败原因、耗时、关联指标

3. **MQ多协议适配层**
    - 定义统一的通用接口：`连接验证、状态查询、消息发送、消息消费、消费状态查询`
    - 分别实现RabbitMQ、Kafka、RocketMQ三个驱动，封装各产品的原生API差异
    - 后续新增MQ产品只需新增驱动，上层逻辑无需改动

4. **数据存储层**
    - 存储测试任务配置、历史测试记录、异常告警日志
    - 留存关键指标（消息延迟、堆积量、消费者数量），支持趋势查询

## 三、核心维度详细设计
### 3.1 维度一：Broker 健康检查
#### 测试目标
验证MQ中间件服务本身是否存活、可连接、运行状态正常，是所有上层测试的基础。

#### 三级检测策略（从浅到深，逐级执行）
| 检测级别 | 检测方式 | 验证目标 | 异常等级 |
|----------|----------|----------|----------|
| 一级：端口可达性 | TCP Socket 连接探测 | 网络层通、端口监听正常 | 严重 |
| 二级：客户端连接 | 原生客户端建立正式连接+认证 | 服务可用、账号密码/权限正常 | 严重 |
| 三级：集群状态 | 调用管理/运维API查询节点状态 | 节点无故障、无核心告警、分区/队列正常 | 警告 |

#### 执行流程
1. 读取任务配置中的MQ类型、地址、端口、认证信息；
2. 执行一级端口探测：向目标IP+端口发起TCP连接，超时时间默认3秒；
    - 连接失败直接判定为「严重异常」，终止后续检测；
    - 连接成功进入下一级。
3. 执行二级连接验证：创建对应MQ的客户端连接，完成认证握手；
    - 连接失败/认证失败判定为「严重异常」，终止后续检测；
    - 连接成功进入下一级，检测完成后关闭连接。
4. 执行三级状态查询：调用管理接口查询节点核心指标；
    - 关键指标异常（如磁盘满、节点离线、分区不同步）判定为「警告」；
    - 所有指标正常判定为「健康」。

#### 主流MQ适配实现
- **RabbitMQ**
  - 端口：默认5672（AMQP）、15672（管理接口）
  - 二级验证：使用AMQP客户端创建Connection
  - 三级状态：调用`GET /api/overview`管理接口，检查节点状态、队列数、连接数
- **Kafka**
  - 端口：默认9092（客户端端口）
  - 二级验证：创建AdminClient连接，执行一次`listTopics()`
  - 三级状态：通过AdminClient查询Broker节点列表、分区健康状态、副本同步状态
- **RocketMQ**
  - 端口：默认9876（NameServer）
  - 二级验证：通过客户端连接NameServer，获取Broker列表
  - 三级状态：调用Dashboard API或`mqadmin clusterList`，检查Broker节点状态、磁盘使用率

#### 判定标准
- 健康：三级检测全部通过
- 警告：一、二级通过，三级存在非致命告警
- 异常：一级或二级检测失败

### 3.2 维度二：生产端连通性测试
#### 测试目标
验证**被测业务系统（作为生产者）**能够正常连接MQ，并成功将消息发送到指定的队列/主题。

#### 两种测试模式（按需选择）
##### 模式A：主动触发式（精准度高，推荐测试环境使用）
通过触发被测系统的业务动作，验证其能否正常产出消息。
1. **前置依赖**：被测系统提供可调用的触发入口（如HTTP测试接口、模拟业务动作的接口），调用后会向指定队列/主题发送一条消息。
2. **执行流程**
    1. 测试系统生成唯一测试标识（如`test_trace_id=xxx`），通过触发入口传递给被测系统；
    2. 被测系统触发业务逻辑，将携带该测试标识的消息发送到MQ；
    3. 测试系统提前作为消费者订阅目标队列/主题，按测试标识过滤消息；
    4. 在超时窗口内等待消息到达，超时时间默认10秒。
3. **判定标准**
    - 连通正常：超时内收到消息，且消息关键字段符合格式约定；
    - 连通异常：超时未收到消息、消息格式错误、被测系统触发接口报错。

##### 模式B：被动监控式（零侵入，推荐生产环境使用）
无需触发业务，通过MQ侧的指标判断生产端是否在正常工作。
1. **前置依赖**：无业务侵入，仅需MQ的查询权限。
2. **执行流程**
    1. 定期调用MQ管理接口，查询目标队列/主题的「最新消息时间」「消息生产速率」「消息增量」；
    2. 对比历史基线，判断生产行为是否符合预期。
3. **判定标准**
    - 正常：统计周期内有持续的新消息产生，速率在正常波动区间；
    - 异常：超过设定阈值（如5分钟）无任何新消息进入队列。
    - 注意：该模式无法区分“业务低峰无消息”和“生产端故障”，适合配合业务规律使用，或作为辅助监控项。

#### 数据隔离方案
- 所有测试消息必须在消息头/属性中携带固定标记：`test_flag=connectivity_check`，与业务消息明确区分；
- 测试消息体采用统一的极简结构，不携带业务数据；
- 条件允许时，为测试单独创建专用队列/主题，与业务队列物理隔离。

### 3.3 维度三：消费端连通性测试
#### 测试目标
验证**被测业务系统（作为消费者）**能够正常连接MQ、拉取消息并完成消费确认。

完全通过MQ管理接口查询消费者状态，不发送任何消息，对业务零影响。
1. **核心验证指标**
    - 消费者在线数：目标消费组/队列的在线消费者数量是否符合预期；
    - 偏移量更新：消费组的提交偏移量是否持续更新；
    - 堆积趋势：队列消息堆积量是否持续增长、无异常堆积。
2. **执行流程**
    1. 定期调用MQ管理接口，拉取目标消费组/队列的上述三个指标；
    2. 对比配置的预期值和基线，判断消费端是否正常运行。
3. **判定标准**
    - 正常：消费者在线数≥预期值，偏移量持续更新，堆积量稳定或下降；
    - 警告：消费者在线数正常，但堆积量缓慢增长；
    - 异常：消费者离线、偏移量长时间停止更新、堆积量持续快速增长。


#### 主流MQ适配实现
- **RabbitMQ**
  - 调用`GET /api/queues/{vhost}/{queue}`，获取`consumers`（消费者数）、`messages_ready`（堆积数）、`message_stats.deliver_get`（消费速率）
- **Kafka**
  - 通过AdminClient查询消费组的`state`（状态）、`members`（成员数）、`lag`（堆积量）、`committed offset`（提交偏移量）
- **RocketMQ**
  - 通过`mqadmin consumerProgress`或Dashboard API查询消费组的连接数、堆积量、最后消费时间

## 四、测试任务配置规范
每个被测系统对应一个测试任务配置，采用结构化配置，示例字段如下：

| 配置项 | 说明 | 示例值 |
|--------|------|--------|
| taskId | 任务唯一标识 | mq_test_order_producer |
| mqType | MQ类型 | RabbitMQ / Kafka / RocketMQ |
| mqAddress | 连接地址 | 10.0.0.1:5672 |
| authInfo | 认证信息（加密存储） | username/password / SASL配置 |
| targetResource | 目标队列/主题、消费组 | queue=order_queue, group=order_consumer_group |
| testDimensions | 启用的测试维度 | ["broker_health", "producer_connect", "consumer_connect"] |
| triggerMode | 生产端触发方式 | http_trigger / passive_monitor |
| triggerConfig | 触发参数（如HTTP接口地址） | url: http://order-system/test/send |
| timeout | 单次测试超时时间 | 10000ms |
| schedule | 调度周期 | 每5分钟执行一次 |
| alertThreshold | 告警阈值 | 连续2次失败触发告警 |

## 五、非功能设计
### 5.1 环境与数据隔离
- 生产环境默认启用被动探测模式，禁止主动发送大量测试消息；
- 所有测试客户端使用独立的客户端ID/消费者组，避免与业务客户端冲突。

### 5.2 安全与权限
- MQ认证信息加密存储，禁止明文配置；
- 生产环境的测试账号仅授予「查询权限」和「测试队列的收发权限」，禁止操作业务数据；
- 所有测试操作留痕，支持审计追溯。
