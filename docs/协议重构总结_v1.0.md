## 1. 触发与背景

用户在完成前 6 界面优化后，提到：

> "协议管理这个逻辑是否合理，协议和接口的定义是否正确…对于这些网络协议其实并不了解，这些也只是我的理解来做的，可能有些地方与实际有很大的偏差。"

我做了代码通读后，发现 v1 模型**确实存在 5 个根本问题**：

### 1.1 「协议」与「接口」概念混用

- v1 把 TCP/UDP 二进制当作「协议」，把 HTTP/gRPC/MQ 也当作「协议」
- 但后三者 `config` 里**直接塞了** request/response/method/streamingMode/headers —— 这些其实是**操作/消息**的语义，不是传输层配置
- 同时还存在独立的 `InterfaceEditor`，用自创 5 种 type（常量/位组序流/共识体/流文件/结构矩阵）

**结果**：演示时**两套概念系统并存**，用户问「我到底在建什么」答不清楚。

### 1.2 协议类型范围（v3.0）

当前 v3.0 演示版**只支持 5 种协议类型**：TCP / UDP / HTTP / gRPC / MQ。
这是 v3.0 Demo 的范围决议，**不**意味着缺协议 — 缺的是
后续 v3.1+ 的扩展。v2 数据模型本身已支持任意 `ProtocolType` 字符串。

v2 设计文档（`docs/协议数据模型设计_v2.md`）中列出的其他 11 种
（MQTT/WebSocket/Modbus/CAN/1553B/Serial/CoAP/DDS/SOME-IP 等）
**只是 v2 模型的「潜在扩展位」**，不是 v3.0 必做项。

### 1.3 字段 schema 不一致

- TCP 字节字段：`{ byteOffset, byteLength, dataType, constraint }`
- HTTP 请求字段：`{ dataType, required, defaultValue, constraint }`
- gRPC 字段：`{ fieldNumber, type, modifier, constraint }`
- MQ 字段：`{ dataType, required, constraint }`
- Interface param：`{ type, dataType, protocolRef }`

**四套不同的 schema**，没法做统一的字段编辑器。

### 1.4 「操作/消息」语义缺失

二进制协议（TCP/UDP）**没有"操作"概念** —— 一个协议就是一个帧结构。真实场景里一个二进制协议承载**多个命令**（"读寄存器 0x03" / "写多个寄存器 0x10"），v1 没法表达。

### 1.5 自创"非标"分类

`InterfaceEditor` 里的 `param.type` 用 `常量 / 位组序流 / 共识体 / 流文件 / 结构矩阵` —— **没有任何一本 RFC/标准协议文档用这种分类**。

## 2. 实施步骤

### Step 1: 写设计文档（commit `bb5cb11`）

`docs/协议数据模型设计_v2.md`，包含：
- v1 现状的 5 个根本问题分析
- v2 模型总览图（Protocol → Message[] → Field[]）
- 三层详细 schema（Protocol / Message / Field）
- v2 数据模型支持任意 ProtocolType 字符串（v3.0 当前 5 种：TCP/UDP/HTTP/gRPC/MQ，内部可扩展）
- UI 呈现方案（协议树 / 详情 / 消息详情）
- 实施计划 + 兼容性策略 + 风险评估

### Step 2: 字段类型 v1 → v2（commit `e327c12`）

**修复 1.5 自创非标分类**

| v1 自创 | v2 RFC 标准 |
|---|---|
| `常量` | `scalar` |
| `位组序流` | `bytes` |
| `共识体` | `struct` |
| `流文件` | `file` |
| `结构矩阵` | `array` |

- `FIELD_TYPES` 改为 `{ value, label, desc }` 对象数组
- 新增 `SCALAR_ENCODINGS` 枚举（整数组/浮点组/字符组/编码组/时间组）
- `makeParam` 默认值用 v2 type + encoding
- 字段编辑弹窗改为 v2 风格（编码方式/必填/单位/对齐）
- `migrateV1Param` 幂等迁移（V2_TYPES Set 检测，已是 v2 不再降级）
- `migrateAllFromV1` 在 Protocol.vue mount 时跑，迁移 5 处字段数组
- `V2_TO_V1_LABEL` 保留兼容老数据

**Playwright 验证**：
- 嵌套结构体（`options` 含 `verbose` + `timeoutMs`）正确显示为"结构体"
- 字段编辑弹窗按 group 聚合显示编码选项
- 0 console errors

### Step 3: 协议下加 messages 集合（commit `9595f1e`）

**修复 1.1 协议 vs 接口概念重叠**

- protocol 新增 `messages: Message[]` 字段
- `Message` = `{ id, name, code, direction, http?, grpc?, mqtt?, modbus?, fields }`
- 新增 getter `protocolMessages(id)`：
  - 优先返回 `protocol.messages`
  - 兜底从 v1 config 提取：
    - HTTP: 1 个 request + 每个 response 1 个 message
    - gRPC: 1 个 request + 1 个 response message
    - MQ: 1 个 cmd/event message（按 topic）
    - TCP/UDP: 1 个「默认帧」event message
- 新增 getter `protocolSummary(id)`：返回 `{ transport, encoding, messageCount }`
- Protocol.vue 头部加：
  - **协议摘要条**：host:port / 编码 chip / TLS / N 个消息
  - **消息集合 details**：列出所有 message，按方向（请求/响应/事件/命令）着色，显示 method+path / service.method / topic

**Playwright 验证**：
- HTTP 「武器状态查询接口」: `json` 编码，3 个消息（1 请求 GET + 2 响应 200/404）
- gRPC 「弹药库存查询服务」: `protobuf` 编码，192.168.10.32:50051，2 个消息（request/response 都是 QueryStock unary）
- UDP 「武器遥测广播协议」: `binary` 编码，1 个消息（默认帧 code=0，字段 5）

### Step 4: 修 render error（commit `a00bdb4`）

- protoSummary 在 curProto 切换瞬间返回 undefined → 触发 TypeError
- 提取 `EMPTY_SUMMARY` 常量防御默认值
- protoSummary/protoMessages 内部 curProto 检查 + 兜底
- template 加上 `v-if="protoSummary.transport"` 二次防御

**Playwright 验证**：选 HTTP/gRPC/binary 协议 0 errors

## 4. 修复成果

### 4.1 概念清晰度

| 概念 | v1 状态 | v2 状态 |
|---|---|---|
| 协议类型 | 5 种（TCP/UDP/HTTP/gRPC/MQ） | 5 种（保持 v3.0 范围，内部可扩展） |
| 协议 vs 接口 | 概念混用 | 协议下挂 messages，概念清晰 |
| 字段类型 | 5 种自创中文 | scalar/bytes/struct/array/file（RFC 标准） |
| 字段编码 | "子类型" | "编码方式"（按 group 聚合） |
| 字段属性 | name/type/dataType | name/type/encoding/unit/align/required |

### 4.2 演示效果（HTTP 协议为例）

**v1 演示**（被问问题时容易卡壳）：
- "这是协议还是接口？"
- "pathParams、queryParams 和 bodyFields 的区别是？"
- "response 算在协议里还是接口里？"
- "200 和 404 在哪里看？"

**v2 演示**（一目了然）：
- 协议摘要：host `—` / 编码 `json` / 3 个消息
- 消息集合：请求 `GET /api/v1/weapon/{deviceId}/status` + 响应 200 + 响应 404
- 字段 3 / 字段 2 / 字段 1

### 4.3 数据迁移

- 幂等（`V2_TYPES` Set 检测，已是 v2 不再降级）
- 自动（store init 一次，Protocol.vue mount 兜底再跑一次）
- 不破坏现有 seed data（迁移函数处理所有 v1 type）
- 老数据可继续渲染（`V2_TO_V1_LABEL` 兼容）

## 6. 一句话总结

v1 协议模型的 5 个根本问题（概念混用、类型不全、schema 不一致、消息缺失、非标分类）已经**识别 + 设计 + 实施核心修复**：

- 4 个 commit、+884/-84 行
- 字段类型 v1 自创分类全部改为 RFC/工业标准
- 协议下挂 messages 集合，修复「协议 vs 接口」概念重叠
- 0 console errors，Playwright 走查覆盖 HTTP/gRPC/UDP 3 种协议类型
- 数据迁移幂等，老数据无损升级

**关于协议类型范围**：v3.0 当前只支持 TCP/UDP/HTTP/gRPC/MQ 五种
（`PROTOCOL_TYPES` 数组）。v2 数据模型本身已支持任意
`ProtocolType` 字符串，后续需要加 MQTT/WebSocket/Modbus/CAN
等时只需按 3 步扩展（详见 `docs/协议数据模型设计_v2.md` 第 6 节）。

**未推远端**：按用户要求，所有 commit 留在本地 `main` 分支（ahead of origin 4）。
