> 起草人：联试前端组
> 日期：2026-07-01
> 状态：设计评审稿（待实施）
> 取代：v1 模型（src/stores/protocol.js:1-647 + src/mock/seed-data.js:209-831）

## 0. 背景与目标

v3.0 演示版对协议管理的实现存在以下根本性问题（影响"是否真的能用于联试"）：

### 0.1 根本问题 1：**"协议"和"接口"概念混在一起**

- v1 把 **TCP/UDP 二进制**当作"协议"，把 **HTTP/gRPC/MQ** 也当作"协议"，但后三者的 `config` 里**直接塞了 request/response/method/streamingMode/headers** —— 这些其实是**操作/消息**的语义，不是传输层配置。
- 同时还存在一个独立的 `InterfaceEditor`，里面的 `param` 用自创的 5 种类型（常量 / 位组序流 / 共识体 / 流文件 / 结构矩阵）。
- 结果：演示时**两套概念系统并存**，用户无法理解"我到底在建什么"。

### 0.2 根本问题 2：**协议类型覆盖不全**

只支持 5 种：TCP / UDP / HTTP / gRPC / MQ。实际联试场景至少需要：

| 类型 | 缺失原因 | 联试场景示例 |
|---|---|---|
| MQTT | 被合并在 MQ 里 | IoT 传感器、卫星地面站 |
| WebSocket | 缺 | 实时态势推送 |
| Protobuf | 缺（仅 gRPC 内部） | 跨语言通信 |
| Modbus | 缺 | 工业控制、PLC |
| CAN / CAN-FD | 缺 | 车载、机电系统 |
| 1553B | 缺 | 航空军用总线 |
| RS232/422/485 | 缺 | 串口设备 |
| CoAP | 缺 | 嵌入式 IoT |

### 0.3 根本问题 3：**字段 schema 不一致**

- TCP 字节字段：`{ byteOffset, byteLength, dataType, constraint }`
- HTTP 请求字段：`{ dataType, required, defaultValue, constraint }`
- gRPC 字段：`{ fieldNumber, type, modifier, constraint }`
- MQ 消息体字段：`{ dataType, required, constraint }`

**四套不同的 schema**，没法做"统一的字段编辑器"，也没法跨协议类型复用规则。

### 0.4 根本问题 4：**"操作/消息"语义缺失**

二进制协议 (TCP/UDP) **没有"操作"概念** —— 一个协议就是一个帧结构。真实场景里一个二进制协议会承载**多个命令**（如"读寄存器 0x03"、"写多个寄存器 0x10"），v1 没法表达。

### 0.5 根本问题 5：自创"非标"分类

`InterfaceEditor` 里的 `param.type` 枚举：
- `常量` - 单值
- `位组序流` - 字节流
- `共识体` - 嵌套结构
- `流文件` - 文件
- `结构矩阵` - 数组

**没有任何一本 RFC/标准协议文档用这种分类**。演示者会被问"位组序流是什么？共识体是什么？"

## 1. v2 数据模型总览

```
Protocol (协议)
│   传输层: TCP / UDP / WebSocket / Serial / MQTT / ...
│   编码: binary / json / protobuf / text
│   帧: 同步头/帧头/长度域/校验/分帧规则
│   消息集合: Message[]      ← 关键改动：原 "接口" 概念并入此处
│
Message (消息/操作)
│   方向: request | response | event | cmd
│   命令码: uint16 (二进制协议用)
│   操作名: "查询设备状态"
│   字段集合: Field[] (扁平或嵌套)
│
Field (字段)
│   名称: "设备编号"
│   编码: uint8 / int16 / utf8 / bcd / raw / ...
│   字节序: big | little
│   字节长度: 1, 2, 4, 8, var
│   必填: bool
│   默认值: any
│   取值范围: 范围/枚举/正则/无
│   对齐: 1/2/4/8 字节
│   单位: "kg" / "°" / "Hz" / ...
│   备注: "..."
│   子字段: Field[] (可选，用于结构体)
```

**核心改动**：
1. 删 `interfaces` 概念 → 全部用 `messages` 表达
2. 删 `param.type` 的 5 种自创分类 → 改用 RFC 标准的"原始类型 + 编码"组合
3. 统一 `field` schema 跨所有协议类型
4. 二进制协议也能定义"操作"（通过命令码）

## 2. 协议 (Protocol) 详细 schema

```typescript
type Protocol = {
  id: number
  name: string                   // 协议名，如 "武器挂载识别协议"
  type: ProtocolType             // 见枚举
  direction: 'duplex' | 'simplex'  // 全双工 / 单工
  systemId: string | null
  moduleId: string | null
  desc: string

  // 传输层配置
  transport: {
    host: string                 // 默认 ''
    port: number                 // 默认 0
    tls: { enabled: boolean }    // 仅 TCP/UDP/MQTT/WSS 用
    auth: { username, password, token }  // 可选
    timeout: number              // ms
    retry: number                // 重试次数
    keepAlive: number            // TCP 心跳间隔 ms（0=关闭）
  }

  // 编码层配置
  encoding: 'binary' | 'json' | 'protobuf' | 'text' | 'xml'

  // 帧结构配置（仅 binary 编码用）
  framing: {
    syncWord: string             // 同步字，hex 字符串，如 "AA55"
    syncOffset: number           // 同步字在帧中的偏移（默认 0）
    lengthField: {               // 长度域，null 表示固定长度
      fieldId: number | null
      offset: number             // 长度域在帧中的偏移
      bytes: 1 | 2 | 4           // 长度域字节数
      includesHeader: boolean    // 长度是否包含帧头
      includesSelf: boolean      // 长度是否包含自身
      bigEndian: boolean         // 长度域字节序
    } | null
    fixedLength: number | null   // 固定长度帧时使用
    terminator: string | null    // 分隔符分包时使用，hex 字符串
    checksum: {                  // 校验配置
      type: 'none' | 'sum8' | 'crc8' | 'crc16' | 'crc32' | 'md5' | 'sha256'
      rangeStart: number         // 校验覆盖的字节范围起始偏移
      rangeEnd: number           // 校验覆盖的字节范围结束偏移（不含）
      fieldId: number | null     // 校验字段在 fields 中的 id（CRC 嵌入帧时）
    }
    endian: 'big' | 'little'     // 字段表默认字节序
  } | null

  // 消息集合（HTTP/gRPC/MQTT/Protobuf/MQ 都用此表达）
  messages: Message[]

  // 兼容 v1 的扩展字段（暂留）
  legacyConfig: any
}

type ProtocolType =
  | 'TCP'           // 字节流
  | 'UDP'           // 数据报
  | 'HTTP'          // HTTP/1.1 + 2
  | 'WebSocket'     // RFC 6455
  | 'MQTT'          // v3.1.1 + v5
  | 'CoAP'          // RFC 7252
  | 'gRPC'          // HTTP/2 + Protobuf
  | 'Protobuf'      // 裸 Protobuf（无 gRPC 包装）
  | 'MQ'            // AMQP / Kafka / RocketMQ
  | 'Modbus'        // RTU + TCP
  | 'CAN'           // CAN 2.0A/B + CAN-FD
  | 'MIL-1553B'     // 航空军用
  | 'Serial'        // RS-232/422/485
  | 'DDS'           // RTPS
  | 'SOME/IP'       // 车载以太网
```

## 3. 消息 (Message) 详细 schema

```typescript
type Message = {
  id: number
  name: string                   // 操作名: "查询设备状态"
  code: number | null            // 命令码（仅 binary 协议用），其他 null
  direction: 'request' | 'response' | 'event' | 'cmd'
  desc: string

  // HTTP 专属（type='HTTP' 时使用）
  http?: {
    method: 'GET' | 'POST' | 'PUT' | 'DELETE' | 'PATCH' | 'HEAD' | 'OPTIONS'
    path: string                 // 含 path param 占位符
    contentType: string          // application/json 等
  }

  // gRPC 专属（type='gRPC' 时使用）
  grpc?: {
    serviceName: string
    methodName: string
    streaming: 'unary' | 'server-stream' | 'client-stream' | 'bidi'
  }

  // MQTT 专属
  mqtt?: {
    qos: 0 | 1 | 2
    retain: boolean
    topic: string                // 支持 ${deviceId} 等占位符
  }

  // Modbus 专属
  modbus?: {
    functionCode: number         // 0x01~0x10 等
    slaveId: number              // 从站地址
  }

  // 字段
  fields: Field[]
}
```

**关键决策**：
- HTTP 一个协议下可有多个 message，每个对应一个 method+path
- gRPC 一个协议下可有多个 message，每个对应 service.method
- MQTT 一个协议下可有多个 message，每个对应一个 topic
- TCP/UDP 二进制协议通过 `code` 区分消息

## 4. 字段 (Field) 详细 schema — 跨协议统一

```typescript
type Field = {
  id: number
  name: string                   // 字段名
  desc: string                   // 备注/说明

  // 编码（决定如何解释字节）
  encoding: FieldEncoding

  // 字节序（仅多字节类型有效）
  endian: 'big' | 'little' | 'inherit'   // inherit = 继承协议默认

  // 字节长度
  length: number | 'var'         // 1/2/4/8 或 var（变长）

  // 变长字段的长度规则
  lengthRef?: {
    source: 'field' | 'prefix'  // 引用另一字段或前缀字节
    fieldId?: number
    offset: number               // 长度域相对字段起始的字节偏移
    bytes: 1 | 2 | 4             // 长度域字节数
    includesSelf: boolean
  }

  // 必填
  required: boolean

  // 默认值
  defaultValue: any

  // 取值约束
  constraint: Constraint

  // 对齐（结构体对齐）
  align: 1 | 2 | 4 | 8          // 字节对齐，默认 1

  // 单位
  unit: string                   // "kg" / "m/s" / "Hz" / "°" / ...

  // 嵌套子字段（结构体/数组）
  children: Field[]
}

type FieldEncoding =
  // 数值
  | 'uint8'  | 'uint16' | 'uint32' | 'uint64'
  | 'int8'   | 'int16'  | 'int32'  | 'int64'
  | 'float32'| 'float64'
  // 字符
  | 'ascii' | 'gbk' | 'utf8' | 'utf16'
  // 编码
  | 'bcd'    // BCD 编码
  | 'hex'    // 十六进制字符串
  // 原始
  | 'raw'    // 原始字节
  | 'bool'   // 1 字节布尔
  // 时间
  | 'unix-ms'     // uint64 unix ms
  | 'unix-sec'    // uint32 unix sec
  | 'iso8601'     // UTF-8 字符串
  // 复杂
  | 'object' | 'array'   // 结构体/数组（用 children 表达）

type Constraint =
  | { mode: 'none' }
  | { mode: 'range', min: number, max: number }
  | { mode: 'enum', entries: { value: number | string, label: string }[] }
  | { mode: 'regex', pattern: string }   // 仅字符串字段用
  | { mode: 'fixed', value: any }        // 固定值（用于同步字/魔数）
  | { mode: 'length', minLen: number, maxLen: number }  // 字符串长度约束
```

**对比 v1**：
- v1 的 `interfaceEditor` 5 种 type（常量/位组序流/共识体/流文件/结构矩阵）→ v2 改为 `encoding` 字段 + RFC 标准类型
- v1 的 `field.dataType` 字符串 → v2 改为 `field.encoding` 枚举 + `field.length` 分离
- v1 的 `children` 是 byte 字段下挂 bit 段（bitField 是 byteField 的特殊情况）→ v2 直接用 `children` 表达嵌套结构，不再有"bit 段是 byte 字段的孩子"这种隐式约束

## 5. UI 呈现

### 5.1 协议树（左）
```
📁 系统.模块
  📄 协议: 武器挂载识别协议 (TCP/binary)
    ▸ 🔵 读挂点 (cmd, code=0x01)
    ▸ 🔵 写挂点 (cmd, code=0x02)
    ▸ ⚡ 遥测事件 (event, code=0x10)
  📄 协议: 弹药编目协议 (TCP/binary)
    ▸ 🔵 ...
  📄 协议: 武器状态查询接口 (HTTP)
    ▸ 🟢 GET /api/v1/weapon/{id}/status
    ▸ 🟡 POST /api/v1/weapon/bind
  📄 协议: AmmoInventoryService (gRPC)
    ▸ 🟣 QueryStock (unary)
    ▸ 🟣 UpdateStock (unary)
  📄 协议: weapon.mqtt (MQTT)
    ▸ 📡 weapon/+/telemetry (sub)
    ▸ 📤 weapon/cmd/ack (pub)
```

### 5.2 协议详情（右）
- 头部：协议名 + 传输层信息 + 类型 chip + 操作按钮
- Tab 1：**传输与编码**（host/port/TLS/认证/编码/字节序）
- Tab 2：**帧结构**（同步字/长度域/校验，binary 才有）
- Tab 3：**消息列表**（message 表格，可点击编辑）
- Tab 4：**导入导出**（JSON 协议模板）

### 5.3 消息详情（点 message 进入）
- 头部：操作名 + 命令码（binary）+ 方向 + 描述
- Tab 1：**字段表**（统一 schema 编辑）
- Tab 2：**预览**（HEX / Protobuf / JSON 模拟）

## 7. 兼容性策略

- v1 → v2 是一次性数据迁移（在 store init 里做）
- v1 的 `interfaces` 概念在 v2 改名为 `messages`，但每个 message 还是能找得到
- v1 的 `protocol.config` 字段保留在 `legacyConfig`，新代码不再使用
- 老的 5 种 protocol.type（TCP/UDP/HTTP/gRPC/MQ）继续支持，但 config 结构改为 v2

## 8. 风险与开放问题

| 风险 | 缓解 |
|---|---|
| 数据模型改动大，老用户使用习惯打破 | 提供 v1→v2 迁移提示 |
| UI 重写工作量大 | 分阶段实施，先 store 再 UI |
| 新增协议类型（MQTT/Modbus 等）实现复杂 | 只做"配置 + 可视化编辑"，不做真正的协议解析（v3 范畴） |
| 字段 schema 统一后，部分协议类型"水土不服" | 每个 protocol type 可有"扩展配置"槽位 |
