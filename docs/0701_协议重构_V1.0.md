## 一、重构背景

### 1.1 问题描述

原系统中的数据规则分类使用了自定义命名（基本类型、原始字节、结构体、数组、文件流），与设计文档《便携式智能联试工具-0418-V7.4.docx》中定义的五类数据规则不一致，导致：

1. **文档与实现不符**: 设计文档明确定义了五类数据规则，但代码使用了不同的命名
2. **概念混淆**: 用户在使用时需要记忆两套术语
3. **专业性不足**: 演示时容易被质疑"位组序流是什么"等问题

### 1.2 设计文档定义（V7.4）

根据设计文档，五类数据规则定义如下：

| 类别 | 定义 | 示例 |
|------|------|------|
| **标量** | 单一数值，可表示整数或实数，不附带维度或结构信息 | 温度、压力、时间间隔 |
| **位组序流** | 连续二进制位序列，强调顺序与位级解析，区别于传统的"字节流" | 底层通信协议、编码载荷 |
| **共识体** | 由多个字段组成的结构化数据块，字段间存在语义或业务上的强关联 | 完整的指令或状态报告 |
| **流文件** | 持久化的二进制或文本文件，以文件整体为操作单元 | 日志、报文存储与回放 |
| **结构矩阵** | 二维表格形式的数据，行列有明确语义 | 批量参数、配置表、测试用例集 |

## 二、重构目标

### 2.1 架构设计

建立清晰的专业层次结构：

```
系统 (System)
  ↓
模块 (Module/Service) - 部署单元，可分布在服务器
  ↓
接口 (Interface/API) - 测试的直接目标，有明确的输入输出
  ↓
协议 (Protocol) - 接口的数据格式定义
  ↓
数据规则 (Data Rules) - 五类基础数据类型
  - 标量 (Scalar)
  - 位组序流 (Bit Stream)
  - 共识体 (Struct/Message)
  - 流文件 (Stream File)
  - 结构矩阵 (Matrix/Table)
```

### 2.2 关键设计原则

1. **接口 ≠ 协议**: 接口是测试动作（请求/响应），协议是数据格式
2. **一个接口可以使用多个协议**: 比如请求用JSON，响应用Protobuf
3. **协议可以复用**: 同一个协议可以被多个接口使用
4. **校验规则独立**: 校验规则用于验证测试结果是否满足定义的响应接口，与协议定义时的数据校验分开

## 四、兼容性处理

### 4.1 向后兼容策略

为确保平滑过渡，系统同时支持新旧类型名：

1. **数据迁移**: 加载时自动将旧类型名映射到新类型名
2. **类型检查**: 所有类型判断逻辑同时检查新旧名称
3. **显示兼容**: 界面显示优先使用新名称，但能正确渲染旧数据

### 4.2 迁移示例

```javascript
// 自动迁移函数
const migrateV1Param = (p) => {
  if (!p) return p
  let targetType = p.type
  // 先尝试旧中文名映射
  if (V1_TO_V2_TYPE[targetType]) targetType = V1_TO_V2_TYPE[targetType]
  // 再尝试中间版本映射
  else if (LEGACY_TO_NEW_TYPE[targetType]) targetType = LEGACY_TO_NEW_TYPE[targetType]
  // 如果还是无效类型，默认为标量
  if (!VALID_CATEGORIES.has(targetType)) targetType = 'scalar'
  return { ...p, type: targetType, /* ... */ }
}
```
